--- LAYER 2 FOUNDATION: ARP POISONING ---

Before Evil Twin (WiFi), the wired equivalent is ARP poisoning.
Same goal: get between victim and gateway. Different mechanism.

HOW IT WORKS:
1. Attacker sends unsolicited ARP reply to victim:
   "192.168.122.1 is-at AA:BB:CC:DD:EE:FF" (attacker's MAC)
2. Victim updates ARP cache — no verification, no authentication
3. All victim traffic now flows to attacker instead of gateway
4. Attacker forwards to real gateway — victim notices nothing

DETECT IT:
ip neigh show | grep 192.168.122.1
Compare MAC against known legitimate gateway MAC.
Unexpected MAC = poisoned cache.

VERIFY LEGITIMATE GATEWAY MAC:
ip neigh show
Expected: 52:54:00:89:4f:85

CAPTURE ARP TRAFFIC:
sudo tcpdump -i enp1s0 -n arp

FLUSH POISONED CACHE:
sudo ip neigh flush all

CONNECTION TO EVIL TWIN:
Evil Twin = ARP poisoning at WiFi layer.
Same attack, different medium.
Attacker gets in position → DNS spoofing → SSL stripping.
ARP poisoning is how "in position" happens on wired networks.


DRILL 10 — Network Attacks: Evil Twin, DNS Spoofing, SSL Stripping
Topic: Network Attacks
Repeat until the full attack chain reasoning is automatic.

--- THE FULL ATTACK CHAIN, IN ORDER ---

1. EVIL TWIN (gaining position):
An attacker broadcasts a WiFi network with the SAME NAME as one your device already
trusts. Devices auto-connect to a recognized SSID with NO WARNING — there's no
certificate, no identity check at the WiFi-connection layer at all. From this point,
ALL your traffic passes through hardware the attacker fully controls, before any TLS
handshake even begins.

2. DNS SPOOFING (redirecting once in position):
Once on the attacker's network, they control DNS responses too. They can answer any
DNS query with whatever IP they choose. BUT: this alone doesn't defeat TLS — a fake
server still can't get a REAL CA-signed certificate for a domain it doesn't control
(proven in Topic 9: subject == issuer is the tell). A spoofed-but-TLS-protected site
still throws a hard browser warning.

3. SSL STRIPPING (the actual credential-theft method):
Since faking TLS fails, the attacker instead prevents TLS from EVER STARTING. The
HTTP-to-HTTPS redirect is itself a plaintext instruction sent by the real server. An
attacker in position can simply intercept that first plaintext request and never let
the redirect through — serving a fake, unencrypted login page instead. No certificate
ever gets checked because no TLS handshake ever begins. Credentials typed into this
page travel in PLAINTEXT, readable by anyone capturing traffic.

--- PROVE IT: READ REAL PLAINTEXT HTTP ON THE WIRE ---
sudo tcpdump -i enp1s0 -n -A tcp port 80 -w ~/http-cleartext.pcap -c 20 &
disown
curl -s -o /dev/null http://neverssl.com    # a real site that deliberately never uses HTTPS
sudo tcpdump -r ~/http-cleartext.pcap -A -n | grep -A 5 "GET /"

Look for the actual "GET / HTTP/1.1" line and surrounding text, sitting in raw,
readable plaintext inside the packet payload. A real login form's username/password
would appear exactly the same way — no decryption needed, just reading the wire.

--- WHAT TLS CATCHES, AND WHAT IT DOESN'T ---
TLS catches: a fake server presenting a self-signed or wrong-domain certificate
(Topic 9 — subject != expected domain, or subject == issuer).
TLS does NOT catch: SSL stripping, because stripping prevents TLS from starting at
all. There's no certificate to check because no handshake ever happens.

--- THE ACTUAL DEFENSE ---
HSTS (HTTP Strict Transport Security) — a mechanism where a site tells your browser
"never connect to me over plain HTTP again, ever" — closes the stripping window after
the FIRST legitimate visit. The real personal defense: notice when a site you know is
normally HTTPS loads as plain HTTP, and don't proceed. Don't click through certificate
warnings. Verify a WiFi network's legitimacy before connecting on unfamiliar networks.

--- PRODUCTION SCENARIO ---
A teammate's credentials are compromised after using public WiFi.
Investigate, don't assume:
1. Did they get a certificate warning and click through it? (TLS identity failure,
   user-bypassed)
2. Was the login page they used actually HTTP, not HTTPS? (SSL stripping — check
   browser history/network logs for the actual scheme used)
3. Did the WiFi network behave unexpectedly (same name, different location/venue)?
   (Evil Twin)
Each answer points to a different root cause and a different fix — don't guess,
verify which link in the chain actually failed.

--- LAYER 2 ADDITION: ARP POISONING (the layer beneath DNS spoofing) ---

CONCEPT:
Before a machine can send a packet to an IP on its local network, it needs to know
that IP's MAC (hardware) address — this mapping lives in the ARP cache. ARP poisoning
means injecting a FAKE mapping, redirecting traffic at the hardware-address layer,
even lower-level than DNS spoofing.

CHECK CURRENT ARP CACHE:
ip neigh show | grep <gateway-ip>

INJECT A FAKE ENTRY (simulation):
sudo ip neigh replace <gateway-ip> dev <iface> lladdr <fake-mac> nud reachable

VERIFIED FINDING (real, hands-on, not theoretical):
A single, one-time fake ARP injection is NOT actually persistent. Tested directly:
injected a fake MAC for the real gateway, then pinged it. The ping succeeded — and
checking the ARP cache immediately after showed the kernel had SILENTLY SELF-HEALED,
replacing the fake entry with the real gateway MAC the moment genuine traffic was
attempted and the fake address failed to respond on the wire.

REAL LESSON: genuine ARP poisoning attacks work by CONTINUOUSLY, REPEATEDLY sending
fake ARP replies faster than the kernel's self-healing can correct them — not a
single static injection. A one-time poison demonstrates the MECHANISM (the cache can
be manipulated) but not the SUSTAINED nature of a real, effective attack. Don't
mistake a successful single injection for proof an attack would actually work in
practice — verify by checking the cache again after real traffic, the same way this
was caught here.

DEFENSE:
Static ARP entries (marking critical gateways as PERMANENT rather than dynamic)
resist this kind of poisoning, since the kernel won't auto-replace a permanent entry
the way it does a dynamic one. On networks where this matters, switches with "Dynamic
ARP Inspection" can also block unauthorized ARP replies at the network hardware level.

--- ALSO LEARNED THIS SESSION: A REAL OPERATIONAL INCIDENT ---
The break script for this section was originally run on the WRONG MACHINE (the host
laptop, not this training VM) due to auto-detecting the wrong interface with no
verification step. Caught by noticing an unfamiliar interface name (wlp0s20f3 instead

any script that modifies networking state should confirm its target environment
before acting, never assume context silently. Also discovered during this incident:
a failed `cd` command does not stop a script/command chain from continuing — it
silently runs subsequent commands in the WRONG directory. Always verify with `pwd`
or `hostname` after any `cd`, especially when juggling multiple machines (host + VM).
