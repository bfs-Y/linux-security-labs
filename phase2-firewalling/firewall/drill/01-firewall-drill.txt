DRILL 05 — Firewall Rules: Reading, Writing, Verifying
Topic: Firewalls (ufw / iptables)
Repeat daily until instant.

--- READ ---
sudo ufw status verbose                          # friendly view: default policy + all rules
sudo iptables -L ufw-user-input -n -v             # raw kernel rules ufw actually generated
sudo iptables -L ufw-before-input -n -v | tail -15  # infrastructure chain: stateful tracking, ICMP, defaults

--- WHAT TO LOOK FOR ---
Default: deny (incoming)        = anything with no matching rule gets dropped
source 0.0.0.0/0                = open to the entire internet — dangerous for sensitive ports
source X.X.X.X/24                = restricted to a specific subnet — what you want for SSH, admin ports
ctstate RELATED,ESTABLISHED      = replies to connections you already initiated, always allowed
pkts/bytes = 0                   = rule exists but hasn't matched any traffic yet (not necessarily wrong)

--- WRITING A RULE: BUILD THE SENTENCE FIRST ---
"Traffic from <SOURCE>, destined for port <PORT>, protocol <TCP/UDP>, should be <ALLOW/DENY>"
Then translate:
sudo ufw allow from <SOURCE_SUBNET> to any port <PORT> proto tcp

--- BREAK ---
sudo ufw allow 9090/tcp                  # opens port to entire internet, no source restriction
sudo ufw status verbose | grep 9090      # confirm "Anywhere" — this is the dangerous state

--- FIX ---
sudo ufw delete allow 9090/tcp
sudo ufw allow from 192.168.122.0/24 to any port 9090 proto tcp
sudo ufw status verbose | grep 9090                      # confirm subnet-restricted
sudo iptables -L ufw-user-input -n -v | grep 9090         # confirm at kernel level too

--- DROP vs REJECT, KNOW THE DIFFERENCE ---
DROP   = silent discard, attacker gets no response, doesn't know if port is closed or host is dead (slower recon for them)
REJECT = active refusal sent back, attacker instantly knows something is alive (faster but gives away information)
Hardened production systems default to DROP for unsolicited traffic — invisibility over politeness.

--- PRODUCTION SCENARIO ---
Security audit flags a port open to 0.0.0.0/0 that should be internal-only.
Step 1: sudo ufw status verbose — confirm the over-permissive rule exists
Step 2: identify the correct source range your legitimate traffic actually needs
Step 3: delete the wide-open rule, add the properly scoped one
Step 4: verify at BOTH layers — ufw status AND raw iptables — never trust one view alone
Step 5: test the live behavior, don't just read the config

--- ENABLE / DISABLE: READ-ONLY AWARENESS, NO DESTRUCTIVE TEST ---
sudo systemctl status ufw --no-pager     # confirms ufw is set to start on boot (enabled/disabled at systemd level)
sudo ufw enable                          # activates rule enforcement
sudo ufw disable                         # DANGER: removes ALL enforcement, does not just "open SSH"

CRITICAL MISCONCEPTION TO AVOID:
Disabling ufw does NOT cut off your current session. Existing connections (ESTABLISHED state)
keep flowing because there's no rule left to block anything. Disabling ufw means EVERYTHING
becomes unconditionally allowed — every port, every service, silently. Nothing visibly breaks,
which is exactly why this is dangerous: no immediate symptom, but all protection is gone.

NEVER run `ufw disable` on a production box without:
1. Confirming what other protection exists (security groups, network ACLs, segmentation)
2. A clear, time-boxed reason and a plan to re-enable
3. Verification immediately after re-enabling that prior rules are intact

--- NAT / MASQUERADE: WHY CONTAINER TRAFFIC CAN REACH THE INTERNET AT ALL ---

CONCEPT:
NAT (Network Address Translation) rewrites the IP address in a packet as it crosses a
device, so two networks that couldn't otherwise talk directly still can. Same thing your
home router does for every device on your private network reaching the real internet.
MASQUERADE is the dynamic variant — used when the rewriting device doesn't have one fixed
known IP to use, so it figures out the correct source address automatically per-interface.

THE MECHANISM:
A container's private IP (e.g. 172.17.0.2) means nothing outside this machine. When it sends
a packet to the real internet, MASQUERADE rewrites the source to the host's real IP and a
chosen port, and tracks that mapping in the connection tracking table (ctstate) you've already
read in iptables output all module. When the reply comes back addressed to that host IP/port,
the kernel looks up the tracking table and hands it back to the correct container.

READ:
sudo iptables -t nat -L -n -v | grep MASQUERADE   # the actual NAT rule, with hit counters

PROVE IT (hands-on, not just reading):
1. sudo docker inspect <container> | grep IPAddress     # note the container's private IP
2. sudo tcpdump -i enp1s0 -n host <some-real-destination-ip>   # capture on the REAL interface
3. sudo docker exec <container> curl http://example.com         # trigger traffic from inside
4. Read the capture: source IP should be the HOST's real IP, NOT the container's private IP
5. sudo iptables -t nat -L -n -v | grep MASQUERADE   # confirm packet/byte counters increased

KEY LESSON:
The container believes it's talking directly to example.com as 172.17.0.2.
example.com believes it's talking to your host's real IP.
Both are correct from their own point of view — NAT is the invisible translator making
both versions of the truth simultaneously valid. This is not theoretical; it's provable
with tcpdump and a non-zero counter, the same evidence standard as every other topic.
